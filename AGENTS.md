# AGENTS.md

## Project Overview

This is a simple technical portfolio website for an aspiring Data Engineer.

The goal is not to showcase advanced frontend skills. The goal is to help recruiters understand the candidate's data engineering ability, technical thinking, project experience, and communication skills.

## Primary Goal

Make the portfolio:
- Clear
- Fast
- Professional
- Easy to read
- Easy to update
- Focused on data engineering projects and case studies

## Tech Direction

Prefer:
- Astro
- Tailwind CSS
- Markdown or MDX
- Static pages
- Simple TypeScript data files

Avoid:
- Complex frontend architecture
- Heavy animations
- Unnecessary JavaScript
- Overly decorative UI
- Large component libraries

## Design Direction

Use a minimal technical style:
- Clean layout
- Strong typography
- Good spacing
- Subtle borders
- Simple cards
- Clear section hierarchy
- Mobile-friendly design

The site should feel like an engineering profile, not a flashy frontend portfolio.

## Data Engineering Focus

Prioritize content about:
- ETL/ELT pipelines
- SQL
- Python
- Data modeling
- Data warehouses
- Batch processing
- Orchestration
- dbt
- Airflow
- Docker
- Cloud platforms
- Data quality
- Monitoring
- Reliability
- Analytics-ready datasets

## Project Presentation

Each project should explain:
- What problem it solves
- What data sources it uses
- How the pipeline works
- What tools were used
- What data model was created
- How data quality was handled
- What the outcome was

Avoid vague project descriptions.

Bad:
“Built a data pipeline using Python.”

Good:
“Built a batch ETL pipeline that ingests raw transaction data, validates schemas, transforms it into dimensional models, and produces daily revenue metrics.”

## Case Study Standard

Each major project should have a case study with:
- Problem
- Data sources
- Architecture
- Tech stack
- Pipeline design
- Data model
- Data quality checks
- Challenges
- Outcome
- Links

## Code Style

Keep code simple:
- Prefer static rendering
- Prefer data-driven components
- Keep content outside UI components
- Avoid unnecessary abstractions
- Use semantic HTML
- Keep accessibility in mind

## Validation

Before finishing:
- Run the available build command
- Run lint/typecheck if configured
- Fix errors
- Clearly mention if any command is unavailable

## Completion Criteria

A task is complete when:
- The portfolio is readable and professional
- Data engineering content is emphasized
- Projects are easy to update
- Case studies use markdown or MDX
- The site builds successfully
- Remaining manual content placeholders are clearly marked
